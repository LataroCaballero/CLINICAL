## Summary

Phase 2.1 is a tightly scoped backend bug-fix. The root cause is fully understood from reading the actual code: `createContacto` in `pacientes.controller.ts` (line 109) reads `profesionalId` directly from `req.user.profesionalId`. The JWT strategy (`jwt.strategy.ts`) sets this field to `usuario.profesional?.id ?? null`. SECRETARIA users have no linked `Profesional` row, so the field is `null`. The service then passes `null` to `tx.contactoLog.create()`, which hits the Prisma FK constraint on `ContactoLog.profesionalId String` (non-nullable, references `Profesional`), resulting in a DB error.

The fix is a single controller action: before calling `pacientesService.createContacto`, if `req.user.profesionalId` is null, perform a Prisma lookup of the patient record to retrieve its `profesionalId`, and use that as the fallback. The service itself does not need to change — it already accepts `profesionalId` as a parameter and passes it directly to Prisma.

No migration, no schema change, no frontend change is needed. The fix is entirely in `pacientes.controller.ts` inside the `createContacto` handler.

**Primary recommendation:** Inject `PrismaService` (already injected in the controller), fetch `paciente.profesionalId` when `req.user.profesionalId` is null, throw a `BadRequestException` if neither is available, and pass the resolved ID to the service.

## Architecture Patterns

### Pattern 1: Controller-level FK resolution before service call

**What:** The controller is already the right place to enrich request data before delegation to the service. The service contract (`createContacto(pacienteId, profesionalId, dto)`) is clean — it expects a resolved `profesionalId`. Resolution logic belongs in the controller, not the service.

**When to use:** When the service has a clean contract requiring a non-null value that can only be resolved from the DB in certain caller roles.

**Example (current broken code):**
```typescript
// pacientes.controller.ts — line 104-111
@Post(':id/contactos')
createContacto(
  @Param('id') pacienteId: string,
  @Body() dto: CreateContactoDto,
  @Req() req: any,
) {
  const profesionalId = req.user?.profesionalId as string; // null for SECRETARIA
  return this.pacientesService.createContacto(pacienteId, profesionalId, dto);
}
```

**Example (fix pattern):**
```typescript
@Post(':id/contactos')
async createContacto(
  @Param('id') pacienteId: string,
  @Body() dto: CreateContactoDto,
  @Req() req: any,
) {
  let profesionalId = req.user?.profesionalId as string | null;

  if (!profesionalId) {
    // SECRETARIA path: resolve from patient record
    const paciente = await this.prisma.paciente.findUnique({
      where: { id: pacienteId },
      select: { profesionalId: true },
    });
    if (!paciente) throw new NotFoundException('Paciente no encontrado');
    profesionalId = paciente.profesionalId;
  }

  if (!profesionalId) {
    throw new BadRequestException(
      'No se pudo determinar el profesional asociado al paciente',
    );
  }

  return this.pacientesService.createContacto(pacienteId, profesionalId, dto);
}
```

### Pattern 2: Existing PrismaService injection in controller

**What:** `PrismaService` is already injected in `PacientesController` (line 31: `private readonly prisma: PrismaService`). The controller already uses it directly for objecion operations. This means the fallback query can be added with zero infrastructure changes.

**Key observation:** `findUnique` with `select: { profesionalId: true }` is a minimal indexed query (PK lookup). No performance concern.

### Anti-Patterns to Avoid

- **Modifying the service signature:** Do not add a `fallback` parameter or role-awareness inside `createContacto` service method. The service is clean and must stay clean.
- **Changing the JWT payload:** Do not try to add `profesionalId` to the SECRETARIA JWT — SECRETARIA users legitimately have no `profesionalId`.
- **Silent null passthrough:** Do not allow `null` to reach Prisma. Always throw a meaningful error if the patient itself has no assigned professional.

## Common Pitfalls

### Pitfall 1: Patient with no profesionalId assigned
**What goes wrong:** Some patients may genuinely have `profesionalId = null` (e.g., newly created without professional assignment). The fallback will also return null, and passing it to Prisma still fails.
**Why it happens:** `Paciente.profesionalId` is `String?` (nullable) in schema.
**How to avoid:** After the DB lookup, check if `paciente.profesionalId` is still null, and throw `BadRequestException('No se pudo determinar el profesional asociado al paciente')`.
**Warning signs:** The fix works for most cases but still breaks on unassigned patients — test this edge case explicitly.

### Pitfall 2: Async controller method requires `async`/`await`
**What goes wrong:** The current `createContacto` handler is synchronous (returns a Promise from the service). Adding an `await this.prisma.paciente.findUnique()` requires making the method `async`.
**Why it happens:** Developer forgets to add `async` keyword.
**How to avoid:** Mark handler `async createContacto(...)` and `await` the Prisma call.
**Warning signs:** TypeScript will infer return type incorrectly if `async` is missing; the Prisma call returns a Promise that gets returned as-is.

### Pitfall 3: NotFoundException vs BadRequestException semantics
**What goes wrong:** Throwing `NotFoundException` when the patient is found but has no profesionalId set — these are two distinct cases.
**Why it happens:** Copy-paste from `findOne`.
**How to avoid:**
- `NotFoundException` → patient ID not found in DB
- `BadRequestException` → patient found but has no profesionalId assigned (business constraint)

## Open Questions

1. **Should `getListaAccion` also be fixed for SECRETARIA?**
   - What we know: `GET /pacientes/lista-accion?profesionalId=X` expects an explicit `profesionalId` query param. SECRETARIA could pass a `profesionalId` as a query param when calling the endpoint.
   - What's unclear: Whether the frontend currently passes `profesionalId` for SECRETARIA users, or relies on the JWT's `profesionalId`.
   - Recommendation: Out of scope for this phase (plan 02.1-01 only targets the POST contacto endpoint). The list endpoint accepts an explicit param and SECRETARIA can pass the professional's ID when needed.
   - Impact if deferred: LOW — ACCION-03 is about registering the interaction from the action list, not about viewing it. Fix POST first.

2. **Should `NotFoundException` be thrown in the controller if `paciente` is not found, or should we rely on the service's own check?**
   - What we know: `pacientesService.createContacto` does NOT check if the patient exists before inserting — it only validates via Prisma FK. The controller's new DB lookup naturally acts as a precheck.
   - What's unclear: Whether to keep the explicit check in the controller or let Prisma's FK error propagate.
   - Recommendation: Keep the explicit `NotFoundException` in the controller — it provides a clear HTTP 404 before reaching the service, consistent with codebase patterns (e.g., `findOne` method in service).
