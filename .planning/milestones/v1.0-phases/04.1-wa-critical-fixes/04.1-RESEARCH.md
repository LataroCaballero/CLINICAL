# Phase 4.1: WA Critical Fixes — Research

**Researched:** 2026-02-28
**Domain:** NestJS controller null-guard, frontend modal UX fix, environment variable documentation
**Confidence:** HIGH — all three gaps are fully diagnosed with exact file locations, root causes, and a proven fix pattern from Phase 2.1

---

## Summary

Phase 4.1 closes three specific, well-defined integration gaps identified in the v1.0 milestone audit. None require schema migrations, new modules, or new dependencies. Every change is surgical: one controller null-guard, one frontend modal decision, and one documentation update.

**MISSING-1** (WA-01): `SendWAMessageModal.tsx` email channel calls `POST /pacientes/:pacienteId/enviar-email` — a route that does not exist. The correct email endpoint for presupuestos is `POST /presupuestos/:id/enviar-email` (PresupuestosController, Phase 3), but `SendWAMessageModal` carries no presupuesto context. Resolution: remove the email channel tab from `SendWAMessageModal`; the email path is already fully served by `EnviarPresupuestoModal` which has the correct presupuesto context and endpoint.

**MISSING-2** (WA-01, WA-05): Every endpoint in `WhatsappController` does `req.user.profesionalId as string` with no null check. SECRETARIA JWT has `profesionalId = null` (confirmed in `jwt.strategy.ts` line 43). This causes `NotFoundException` on all send/thread/retry calls and silently returns an empty unread map. Phase 2.1 fixed the identical bug in `pacientes.controller.ts` using an async DB lookup fallback (`paciente.profesionalId`). The same pattern applies here, but with a nuance: WA operations require the WABA to be registered under a profesional — the fallback must look up the profesionalId from the relevant entity (paciente or mensaje).

**MISSING-3** (PRES-04): `BACKEND_URL` is used in `whatsapp.controller.ts:131-134` to build the public PDF URL that Meta fetches. It falls back to `localhost:3001` in production. The env var is undocumented (not in CLAUDE.md, not in any `.env.example`). Fix: add documentation to CLAUDE.md and create `backend/.env.example` with the variable shape.

**Primary recommendation:** Three targeted fixes in one plan — MISSING-2 backend null-guard, MISSING-1 frontend modal simplification, MISSING-3 documentation. No new libraries, no migrations, no module changes.

## Architecture Patterns

### Pattern 1: SECRETARIA profesionalId Null-Guard (established in Phase 2.1)

**What:** When SECRETARIA JWT has `profesionalId = null`, resolve the correct `profesionalId` from the patient record stored in DB. Applied in `pacientes.controller.ts:createContacto` (lines 105–123).

**The JWT reality** (from `jwt.strategy.ts:43`):
```typescript
profesionalId: usuario.profesional?.id ?? null,
```
SECRETARIA users have no linked `Profesional` row, so `profesionalId` is always `null` in the JWT payload.

**Established fix pattern (Phase 2.1, `pacientes.controller.ts:100-123`):**
```typescript
async createContacto(
  @Param('id') pacienteId: string,
  @Body() dto: CreateContactoDto,
  @Req() req: any,
) {
  let profesionalId = req.user?.profesionalId as string | null;

  if (!profesionalId) {
    // SECRETARIA path: resolve profesionalId from the patient record
    const paciente = await this.prisma.paciente.findUnique({
      where: { id: pacienteId },
      select: { profesionalId: true },
    });
    if (!paciente) throw new NotFoundException('Paciente no encontrado');
    profesionalId = paciente.profesionalId;
  }

  if (!profesionalId) {
    throw new BadRequestException(
      'No se pudo determinar el profesional asociado al paciente',
    );
  }

  return this.pacientesService.createContacto(pacienteId, profesionalId, dto);
}
```

**Adaptation for WhatsappController:** The controller currently accesses PrismaService only via `WhatsappService`. To apply the null-guard in the controller, `PrismaService` must be injected directly into `WhatsappController` (same as `PacientesController`). Since `PrismaModule` is `@Global()` (confirmed by comment in `whatsapp.module.ts:12`), no module import change is needed — just inject `PrismaService` in the controller constructor.

**Endpoints affected in WhatsappController:**
| Endpoint | Lookup source | Fallback key |
|----------|--------------|--------------|
| GET /whatsapp/templates | No pacienteId — needs profesionalId for WABA config | Look up via any route param available; if none, throw informative error |
| GET /whatsapp/mensajes/:pacienteId | `pacienteId` param | `paciente.profesionalId` |
| POST /whatsapp/mensajes | `dto.pacienteId` | `paciente.profesionalId` |
| POST /whatsapp/mensajes/free-text | `dto.pacienteId` | `paciente.profesionalId` |
| GET /whatsapp/unread | No pacienteId | Cannot resolve without profesionalId — return empty `{}` gracefully |
| POST /whatsapp/mensajes/:id/retry | mensajeId param | `mensaje.profesionalId` directly from DB record |
| POST /whatsapp/presupuesto/:presupuestoId/send | `presupuestoId` + `body.pacienteId` | `paciente.profesionalId` |

**Key insight for `getUnreadCounts`:** With `profesionalId = null`, current code returns `{}` silently (empty object, no exception). The fix here is to return `{}` explicitly when profesionalId is null — no DB lookup needed. Keeps SECRETARIA seeing 0 unread badges vs crashing.

**Key insight for `getTemplates`:** Templates are fetched from Meta via WABA config which is stored per-profesional. A SECRETARIA has no profesionalId — there is no natural fallback. Options: (A) reject with `BadRequestException('SECRETARIA debe seleccionar un profesional')`, (B) look up the first active WABA in the system (not multi-tenant-safe). Recommendation: option A with a clear message. Templates are only needed for the send modal, and the SECRETARIA flow can pre-fetch the profesionalId from the patient before opening the modal — or the frontend can pass it as a query param.

### Pattern 2: Frontend Modal Email Channel Removal (MISSING-1)

**What:** Remove the email channel toggle from `SendWAMessageModal.tsx`. The modal's sole purpose should be WA template send. Email send of presupuestos belongs to `EnviarPresupuestoModal` which has the correct presupuesto context and connects to `POST /presupuestos/:id/enviar-email`.

**Current state in `SendWAMessageModal.tsx`:**
- Lines 24, 42: `type Channel = 'whatsapp' | 'email'`; `useState<Channel>('whatsapp')`
- Lines 55-75: email branch calls `api.post('/pacientes/${pacienteId}/enviar-email', ...)` → always 404
- Lines 111-136: channel toggle UI (WA + Email buttons)
- Lines 192-207: email confirmation/warning blocks

**Minimal fix:** Remove `Channel` type, `channel` state, email branch in `handleConfirm`, toggle UI, email confirmation blocks, `Mail` import, `isSendingEmail` state. Keep: template list, WA send logic, WA styling.

**Alternative (create the missing endpoint):** Create `POST /pacientes/:id/enviar-email` accepting `{ subject, body }` via Nodemailer. This would require: new endpoint in PacientesController, new DTO, email service injection into PacientesModule. More invasive for less value — SECRETARIA can already send email via presupuesto flow. **Recommendation: remove email channel tab** (Option A per audit).

### Pattern 3: Environment Variable Documentation (MISSING-3)

**What:** `BACKEND_URL` is used in `whatsapp.controller.ts:131-134` to build the public PDF URL for Meta. It is not documented in CLAUDE.md or any `.env.example`.

**Current code:**
```typescript
const backendUrl =
  process.env.BACKEND_URL ??
  process.env.NEXT_PUBLIC_API_URL ??
  'http://localhost:3001';
const pdfPublicUrl = `${backendUrl}/presupuestos/public/${presupuestoId}/pdf`;
```

**Fix scope:**
1. Add `BACKEND_URL=https://api.yourdomain.com` to `backend/.env.example` (create file with all required vars)
2. Add `BACKEND_URL` to CLAUDE.md under "Environment Variables" → Backend section
3. Optionally add a startup warning log if `BACKEND_URL` is not set (similar to EncryptionService dev-key warning)

**Note:** `NEXT_PUBLIC_API_URL` is a frontend env var. Using it as a backend fallback is technically wrong (it would require it to be set in the backend process env, which it is not). The fallback chain should be: `BACKEND_URL ?? 'http://localhost:3001'` with a clear warning log.

### Anti-Patterns to Avoid

- **Injecting WhatsappService into PacientesController** to share the lookup: creates circular dependency risk. The controller should use `PrismaService` directly for the null-guard lookup — same as the Phase 2.1 fix.
- **Global profesionalId resolution helper**: Premature abstraction for three endpoints. Keep null-guard inline per controller following established pattern.
- **Creating /pacientes/:id/enviar-email endpoint**: Adds tech debt without value — email is already covered for presupuestos; creating a freeform email endpoint for templates is not a real use case.

## Common Pitfalls

### Pitfall 1: PrismaService Injection in WhatsappController
**What goes wrong:** Adding `PrismaService` to `WhatsappController` constructor without confirming it is globally available.
**Why it happens:** Developer assumes explicit module import is needed.
**How to avoid:** `PrismaModule` is `@Global()` — confirmed by the comment in `whatsapp.module.ts:12`. Just add `private readonly prisma: PrismaService` to the constructor; no module change required.
**Warning signs:** TypeScript DI error at startup: `Nest can't resolve dependencies of WhatsappController`.

### Pitfall 2: Applying Null-Guard Only to Some Endpoints
**What goes wrong:** Fixing `sendMessage` but missing `getThread`, `retryMessage`, `sendFreeText`, `sendPresupuesto`.
**Why it happens:** Developer scans only the most-used endpoints.
**How to avoid:** Audit all 8 endpoints in `WhatsappController`. Each one passes `req.user.profesionalId as string` to a service method that calls `getDecryptedConfig(profesionalId)`, which throws `NotFoundException` on null.
**Warning signs:** SECRETARIA can send but cannot view thread, or can retry but cannot see unread counts.

### Pitfall 3: getTemplates Null-Guard Design
**What goes wrong:** Attempting to guess the profesionalId for SECRETARIA in `getTemplates` using system-wide lookup (first WABA found).
**Why it happens:** Developer wants SECRETARIA to see templates.
**How to avoid:** Return `BadRequestException` with a clear message. SECRETARIA always opens the send modal from a patient context — the frontend should carry the profesionalId forward, or the templates endpoint can accept an optional `profesionalId` query param. Do not make a system-wide WABA guess.

### Pitfall 4: Removing Email Channel Breaks Type References
**What goes wrong:** Removing `Channel` type and `email` state leaves dangling references if `pacienteEmail` prop is used elsewhere.
**Why it happens:** Quick delete without scanning all usages.
**How to avoid:** Check all consumers of `SendWAMessageModal` for the `pacienteEmail` prop — it can be kept on the type signature for backward compatibility but simply not rendered.

### Pitfall 5: NEXT_PUBLIC_API_URL Fallback
**What goes wrong:** Documenting `NEXT_PUBLIC_API_URL` as the BACKEND_URL fallback in documentation.
**Why it happens:** Current code uses it as a fallback, so it looks intentional.
**How to avoid:** The fallback chain is wrong — `NEXT_PUBLIC_API_URL` is a frontend-only env var. In backend runtime it is undefined unless explicitly set. Document only `BACKEND_URL` as the canonical var; note that the `NEXT_PUBLIC_API_URL` fallback in the code is a dev convenience, not production-ready.

async sendMessage(@Req() req: any, @Body() dto: SendWaMessageDto) {

## Open Questions

1. **getTemplates null-guard strategy**
   - What we know: SECRETARIA cannot call this endpoint without a profesionalId — the service calls `getDecryptedConfig(profesionalId)` which needs the WABA config registered under a specific profesional
   - What's unclear: Does the frontend always open `SendWAMessageModal` from a patient context where the profesionalId could be inferred? If yes, templates could accept `?profesionalId=` query param instead of reading from JWT
   - Recommendation: For Phase 4.1, throw `BadRequestException('SECRETARIA debe estar asociada a un profesional para listar templates')`. If the frontend use case requires it, a future plan can add `?profesionalId=` query param. The email channel removal (MISSING-1) reduces the urgency since templates are only used for WA sends.

2. **WhatsappController PrismaService injection — no module change needed?**
   - What we know: `whatsapp.module.ts:12` has comment `// PrismaModule is @Global() — PrismaService is available without explicit import`; the comment was placed there when building Phase 1 knowing future phases would need Prisma
   - What's unclear: Whether the comment accurately reflects the actual global module registration
   - Recommendation: Verify `PrismaModule` is exported globally in `app.module.ts` before assuming. If not global, add it to the `imports` array of `WhatsappModule`.
