**Section 1: Certificates (Per-Tenant / Per-CUIT)**
- X.509 certificate types (self-signed vs AFIP CA)
- Homologacion vs production: WSASS vs "Administrador de Certificados Digitales"
- Storage approach: model after `EncryptionService` AES-256-GCM pattern from `backend/src/modules/whatsapp/crypto/encryption.service.ts`. Store `certPemEncrypted` and `keyPemEncrypted` as `iv:authTag:ciphertext` strings. Never return in API responses.
- Fields to add to `Profesional` or a dedicated `ConfiguracionAFIP` model in v1.2: `certPemEncrypted`, `keyPemEncrypted`, `certExpiresAt` (warn before expiry)
- Provisioning flow: CSR generation → AFIP portal → download cert → ADMIN uploads cert+key → system encrypts and stores
- Key rotation: store `certExpiresAt`; warn 30 days before expiry

**Section 2: WSAA (Authentication)**
- Endpoints: Homologacion `https://wsaahomo.afip.gov.ar/ws/services/LoginCms`, Produccion `https://wsaa.afip.gov.ar/ws/services/LoginCms`
- Full TRA XML structure with all required elements (`uniqueId`, `generationTime`, `expirationTime`, `service`)
- CMS/PKCS#7 signing: explain that Node.js `crypto` does not natively produce PKCS#7 SignedData. Document TWO options:
  - **Primary (recommended):** `openssl smime -sign` subprocess — no new npm dependency; requires openssl binary on server
  - **Alternative:** `node-forge` npm package — stays in-process, adds dependency
  - Flag as v1.2 implementation decision
- Token extraction from `LoginTicketResponse` XML: `<token>` and `<sign>` from `<credentials>`
- Token caching: cache `{token, sign, expiration}` per CUIT+service. Refresh when `now >= expiration - 5min`. Use a mutex/lock to avoid concurrent renewals per CUIT.
- 12h token lifetime — server MUST use NTP sync (clock skew causes `"Fecha de generación inválida"` rejection)
- Include Node.js TypeScript skeleton for `buildTRA()` (from 08-RESEARCH.md code examples)

**Section 3: WSFEv1 (Electronic Invoicing)**
- Endpoints: Homologacion `https://wswhomo.afip.gov.ar/wsfev1/service.asmx`, Produccion `https://servicios1.afip.gov.ar/wsfev1/service.asmx`
- Key methods: `FEDummy` (health check), `FECompUltimoAutorizado`, `FECAESolicitar`, `FECAEASolicitar`, `FECAEAInformar`, `FEParamGetTiposMonedas`
- Complete list of mandatory request fields for `FECAESolicitar` (all fields from 08-RESEARCH.md Phase Requirements section)
- `CondicionIVAReceptorId` → Prisma enum mapping table (all 11 values with AFIP integer IDs)
- `MonId` mapping: `ARS` → `"PES"`, `USD` → `"DOL"` (do not store AFIP codes — map at submission time)
- **CRITICAL — Advisory lock for CAE sequencing**: Explain that concurrent requests MUST be serialized. Recommended: `SELECT pg_advisory_xact_lock(hashtext(profesionalId || ptoVta || cbteTipo))` wrapping `FECompUltimoAutorizado → build request → FECAESolicitar`. Without this, concurrent submissions get duplicate sequence errors.

**Section 4: CAEA (Contingency Mode)**
- What CAEA is: authorization code obtained in advance, valid for a 15-day billing period
- Restriction from June 2026 (RG 5782/2025): CAEA is contingency-only. Primary flow MUST be CAE. Document this prominently.
- Pre-request timing: within 5 calendar days before period start. Recommend a cron job at `periodo - 5 days`
- Contingency flow: (1) pre-request CAEA, (2) when CAE endpoint fails use cached CAEA, (3) inform via `FECAEAInformar` within 30 days after period ends
- `FECAEASolicitar` key fields: `Periodo` (YYYYMM), `Orden` (1=first half, 2=second half)
- Production reliability: AFIP has maintenance windows and fiscal year-end congestion. Always maintain a pre-requested CAEA for the current period as insurance.

**Section 5: RG 5616/2024**
- Published: December 18, 2024. In force: April 15, 2025 (non-exclusivity until March 31, 2026 — invoices failing the new fields are rejected from April 1, 2026 onward).
- Mandatory new fields in WSFEv1: `CondicionIVAReceptorId` (see Section 3 table) and `MonCotiz` (BNA selling rate for foreign currency)
- Impact on this codebase: `condicionIVAReceptor` and `tipoCambio` + `moneda` added to `Factura` model in migration `facturador_v1` (Phase 8)
- BNA rate source: Banco Nación Argentina tipo vendedor divisa for the business day prior to invoice issuance. Must be fetched from BNA or manually entered; document as v1.2 implementation requirement.

**Section 6: Recommended Library Approach**
- Decision: Raw SOAP/XML — no third-party AFIP library
- Rationale: `afipjs`, `afip-apis`, `afip.js` forks are unmaintained, no TS types, hide WSAA token flow
- What to use:
  - HTTP: `axios` (already in project) for SOAP calls
  - XML parsing: `xml2js` (lightweight, or manual string parsing for simple responses)
  - CMS signing: `openssl smime -sign` subprocess (primary) or `node-forge` (alternative)
  - Encryption: existing `EncryptionService` for cert storage
- SOAP envelope pattern: show minimal SOAP 1.1 envelope for `LoginCms` call
- Implementation location: `backend/src/modules/finanzas/afip/` subdirectory (to be created in Phase 9)
