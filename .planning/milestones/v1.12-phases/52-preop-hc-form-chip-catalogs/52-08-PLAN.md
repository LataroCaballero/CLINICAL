<objective>
Cerrar el gap de UAT Test 13: el envío del link de portal por email siempre falla
("dirección no encontrada o algo así"). Causa raíz (ver
`.planning/debug/portal-link-email-direccion-no-encontrada.md`): el endpoint
`POST /pacientes/:id/portal-link/email` re-llama a `generarPortalLink`, que es
idempotente (D-12) y devuelve `url: null` cuando ya existe un token — lo cual
siempre es el caso cuando la UI de email es visible. El controller corta a
`{ enviado: false }` ANTES de llamar a `portalEmail.enviarLinkPortal`, así que el
email nunca se envía. El frontend muestra un banner genérico engañoso
("Verificá la dirección").

Fix (opción preferida del diagnóstico): desacoplar el envío de la regeneración
del link. El frontend ya tiene la `url` en estado (del paso "Generar"); la pasa al
endpoint. El controller envía esa url provista (tras `setEmailSiFalta` + validación
de email/destinatario), en vez de re-derivarla. D-12 queda intacto: el raw UUID
nunca se persiste ni se reconstruye en el servidor. Como la url ahora viene del
cliente, se valida server-side que sea una url de portal del mismo origen.

Purpose: hacer que el envío por email funcione de forma determinista (Success
Criteria #5 / PREOP-12) y que los banners de error dejen de culpar la dirección.
Output: helper de validación + tests, DTO, controller/service corregidos, hook y
panel frontend actualizados con estados de error diferenciados.
</objective>
