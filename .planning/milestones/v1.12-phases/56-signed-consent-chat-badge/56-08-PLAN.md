<objective>
Populate the `Cirugia.cirugiaCatalogoId` FK (added in Plan 01) so the D-09 consent-resolution chain is actually reachable. Add an optional `cirugiaCatalogoId` to the cirugia create DTO, persist it in `crearTurnoCirugia`, and add a `CirugiaCatalogo` selector to the agenda surgery modal so a professional links a surgery to a catalog entry at creation time.

Purpose: Without this, nothing writes `cirugiaCatalogoId`; the Plan 04 resolver always falls into `SIN_CATALOGO`, the signature canvas never renders, and the phase goal (CONS-03) is unreachable. This closes that gap by making newly-created surgeries linkable to a catalog entry (whose zona carries the vigente consent PDF).

Scope note (edit path): the codebase has NO cirugia edit/update endpoint today — surgeries are created via `POST /turnos/cirugia` (`crearTurnoCirugia`), and `SurgeryAppointmentModal` is create-only; the only `@Patch` routes on turnos operate on turno status (observaciones/cancelar/confirmar/etc.), not on cirugia fields. Per the orchestrator directive ("new/edited surgeries getting the FK is sufficient; existing unlinked ones fall to the D-10 empty-state gracefully"), the create path is the linking mechanism. This plan does NOT invent a new cirugia-edit feature (out of scope, not required for reachability); existing unlinked cirugias resolve to the D-10 empty state.
Output: One DTO field, one service persistence line, and a catalog Select in the surgery modal.
</objective>
