### E2E Flows

#### Flow 1: PENDIENTE patient → surgery booking → CIRUGIA → CRM → LiveTurno (no banner)

1. Patient starts as `flujo = PENDIENTE`
2. Staff books "Consulta para cirugía" turno
3. `crearTurno()` step 5.5 fires: guard passes → `prisma.paciente.update({ flujo: CIRUGIA })` (fire-and-forget)
4. Patient with `flujo = CIRUGIA` included in `getKanban()` via `OR: [{ flujo: CIRUGIA }, { flujo: null }]`
5. Professional opens LiveTurno → `pacienteFlujo = 'CIRUGIA'` in session → banner gate `!== 'PENDIENTE'` → no banner shown

**Status: COMPLETE — no breaks**

#### Flow 2: Walk-in treatment patient → TratamientosTab (not in CRM kanban)

1. New patient created → `flujo = null` (no CRM history)
2. Staff books "Consulta para tratamiento en consultorio" turno
3. `crearTurno()` step 5.5: guard `pacienteCRM.flujo === FlujoPaciente.PENDIENTE` fails (`null !== PENDIENTE`) → no auto-update. Patient stays `flujo = null`
4. **Documented behavior:** `flujo = null` patient IS included in `getKanban()` (legacy preservation). Patient will appear in CRM until classified via LiveTurno banner.
5. `TratamientosTab` filters on `tipoTurno.flujoPaciente === "TRATAMIENTO"` — turno correctly appears in treatments tab

**Status: COMPLETE — behavior documented as intended (LiveTurno banner is the classification driver for new null patients)**

## Tech Debt (Non-Critical)

### Phase 25: Tratamientos Tab

- **Type cast:** `PacienteDetails.tsx` uses `(paciente.flujo as any)` because the component receives `paciente: any`. Not a runtime issue; minor type-safety smell.
- **SUMMARY field name:** Plans 25-01 and 25-02 track requirements under `requirements_satisfied` instead of `requirements_completed`. The GSD summary-extract tool returns empty for these plans. No functional impact but creates inconsistency in tooling.

### Phase 24: LiveTurno Banner

- **Session recovery blind spot:** When `LiveTurnoSyncChecker` recovers a stale session, it sets `pacienteFlujo: null` explicitly. A PENDIENTE patient whose session is recovered will not see the amber banner until they start a fresh session. Conservative design — prevents banner on uncertain state. Documented in VERIFICATION.md.

### Phase 23: Backend Logic

- **Null flujo classification gap:** New patients (flujo=null, no prior CRM history) are NOT auto-classified when booking a treatment. The auto-update guard requires `flujo === PENDIENTE`. New patients must be classified via the LiveTurno banner or PATCH endpoint. By design for legacy preservation, but creates a subtle UX path where a treatment-only patient temporarily appears in the surgery kanban.

### Total: 4 items across 3 phases — no blockers

The milestone goal is fully achieved:

1. **Flujo enum and schema** — `FlujoPaciente { CIRUGIA, TRATAMIENTO, PENDIENTE }` on Paciente, `flujoPaciente` on TipoTurno, 5 canonical appointment types in DB, backfill migration applied
