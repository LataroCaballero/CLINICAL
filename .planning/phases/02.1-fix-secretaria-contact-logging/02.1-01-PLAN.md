<objective>
Fix the null-profesionalId FK crash when a SECRETARIA user logs a contact.

Purpose: LOG-01 and ACCION-03 are blocked for the SECRETARIA role because `req.user.profesionalId` is null in their JWT (SECRETARIA has no linked Profesional row). The `createContacto` handler passes this null directly to the service, which hits a Prisma FK constraint on `ContactoLog.profesionalId` (NOT NULL), producing a 500.

Output: A single controller method change — `createContacto` becomes async, checks if `profesionalId` is null, and if so fetches it from the patient record via the already-injected `PrismaService`. PROFESIONAL users are completely unaffected (fast path, no DB lookup). Service contract is unchanged.
</objective>

    ```typescript
    // Log de Contactos — Registrar una nueva interacción
    @Post(':id/contactos')
    async createContacto(
      @Param('id') pacienteId: string,
      @Body() dto: CreateContactoDto,
      @Req() req: any,
    ) {
      let profesionalId = req.user?.profesionalId as string | null;

      if (!profesionalId) {
        // SECRETARIA path: resolve profesionalId from the patient record
        const paciente = await this.prisma.paciente.findUnique({
          where: { id: pacienteId },
          select: { profesionalId: true },
        });
        if (!paciente) throw new NotFoundException('Paciente no encontrado');
        profesionalId = paciente.profesionalId;
      }

      if (!profesionalId) {
        throw new BadRequestException(
          'No se pudo determinar el profesional asociado al paciente',
        );
      }

      return this.pacientesService.createContacto(pacienteId, profesionalId, dto);
    }
    ```

    Key correctness rules:

    - `profesionalId` must be typed `string | null` before the null check, then `string` after — TypeScript will require the cast after the checks if needed.
    - Do NOT modify `pacientesService.createContacto` — its signature (`pacienteId: string, profesionalId: string, dto`) stays clean.
