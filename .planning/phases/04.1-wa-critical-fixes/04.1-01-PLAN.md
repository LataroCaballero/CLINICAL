    Apply null-guard to each affected endpoint using the Phase 2.1 pattern from `pacientes.controller.ts:105-123`. The pattern:
    ```typescript
    let profesionalId = req.user?.profesionalId as string | null;
    if (!profesionalId) {
      const entity = await this.prisma.<model>.findUnique({ where: { id: param }, select: { profesionalId: true } });
      if (!entity) throw new NotFoundException('<Entity> no encontrado');
      profesionalId = entity.profesionalId;
    }
    if (!profesionalId) {
      throw new BadRequestException('No se pudo determinar el profesional asociado al <entity>');
    }
    ```

    Apply per endpoint as follows:

    **getConfig** (`GET /whatsapp/config`): Keep `ADMIN, PROFESIONAL` only — SECRETARIA has no WABA config; no change needed here.

    **saveConfig** (`POST /whatsapp/config`): Keep `ADMIN, PROFESIONAL` only — no change needed.

    **getTemplates** (`GET /whatsapp/templates`): profesionalId is null for SECRETARIA and there is no natural patient entity to look up. Throw `BadRequestException('SECRETARIA debe estar asociada a un profesional para listar templates')` when profesionalId is null. Keep `SECRETARIA` in `@Auth()` so the request reaches the null-guard (not a 403), but the controller returns a clear 400 error.

    **getThread** (`GET /whatsapp/mensajes/:pacienteId`): Lookup fallback via `this.prisma.paciente.findUnique({ where: { id: pacienteId }, select: { profesionalId: true } })`.

    **sendMessage** (`POST /whatsapp/mensajes`): Lookup fallback via `this.prisma.paciente.findUnique({ where: { id: dto.pacienteId }, select: { profesionalId: true } })`.

    **sendFreeText** (`POST /whatsapp/mensajes/free-text`): Lookup fallback via `this.prisma.paciente.findUnique({ where: { id: dto.pacienteId }, select: { profesionalId: true } })`.

    **getUnreadCounts** (`GET /whatsapp/unread`): Return `{}` immediately when `profesionalId` is null — no DB lookup needed. SECRETARIA sees 0 unread badges without crashing.

    **retryMessage** (`POST /whatsapp/mensajes/:id/retry`): Lookup fallback via `this.prisma.mensajeWhatsApp.findUnique({ where: { id: mensajeId }, select: { profesionalId: true } })` — the mensaje record already has profesionalId on it (stored at send time).

    **sendPresupuesto** (`POST /whatsapp/presupuesto/:presupuestoId/send`): Lookup fallback via `this.prisma.paciente.findUnique({ where: { id: body.pacienteId }, select: { profesionalId: true } })`.

    Also clean up the `sendPresupuesto` fallback chain: remove `process.env.NEXT_PUBLIC_API_URL` from the fallback (it is a frontend-only env var, undefined in backend runtime). Use `process.env.BACKEND_URL ?? 'http://localhost:3001'` and add a `Logger.warn` if `BACKEND_URL` is not set (similar pattern to EncryptionService dev-key warning). Import `Logger` from `@nestjs/common`.
