**`frontend/src/hooks/useAfipConfig.ts`** (follow useWabaConfig.ts pattern exactly):
```typescript
import { useQuery } from "@tanstack/react-query";
import { api } from "@/lib/api";
import type { AfipConfigStatusResponse } from "@/types/afip";

export function useAfipConfig() {
  return useQuery<AfipConfigStatusResponse>({
    queryKey: ["afip-config-status"],
    queryFn: async () => {
      const res = await api.get("/afip-config/status");
      return res.data;
    },
  });
}
```

**`frontend/src/hooks/useSaveCert.ts`**:
```typescript
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { api } from "@/lib/api";
import type { SaveCertRequest, AfipConfigStatusResponse } from "@/types/afip";

export function useSaveCert() {
  const queryClient = useQueryClient();
  return useMutation<AfipConfigStatusResponse, Error, SaveCertRequest>({
    mutationFn: async (data) => {
      const res = await api.post("/afip-config/cert", data);
      return res.data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["afip-config-status"] });
    },
  });
}
```

**`frontend/src/hooks/useSaveBillingConfig.ts`**:
```typescript
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { api } from "@/lib/api";
import type { SaveBillingConfigRequest, AfipConfigStatusResponse } from "@/types/afip";

export function useSaveBillingConfig() {
  const queryClient = useQueryClient();
  return useMutation<AfipConfigStatusResponse, Error, SaveBillingConfigRequest>({
    mutationFn: async (data) => {
      const res = await api.patch("/afip-config/billing", data);
      return res.data;
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["afip-config-status"] });
    },
  });
}
```

**`AfipConfigTab.tsx`** — Two sections, follows WhatsappConfigTab.tsx pattern exactly.

Structure:
1. `useAfipConfig()` for status query
2. `useSaveCert()` mutation
3. `useSaveBillingConfig()` mutation
4. Local state: `isEditingCert` (boolean), `previewData` (null | { cuit, certExpiresAt, ambiente })

**Section 1 — Certificado**:
- When `status.configured && !isEditingCert`: show status view (CUIT, ptoVta, ambiente, certExpiresAt formatted as date string, badge from Pattern 10 of RESEARCH.md). "Actualizar certificado" button sets `isEditingCert = true`.
- When `!status.configured || isEditingCert`: show form with two shadcn Textarea components (label "Certificado PEM (-----BEGIN CERTIFICATE-----...)" and "Clave privada (-----BEGIN PRIVATE KEY-----...)"). Form fields: certPem (Textarea), keyPem (Textarea), ptoVta (Input type=number), ambiente (Select with HOMOLOGACION/PRODUCCION).
- "Vista previa" button (not "Guardar"): calls a local `handlePreview()` that does minimal client-side validation (non-empty fields) and opens a Dialog (preview modal) showing: extracted — wait, note that CUIT extraction is server-side. So "Vista previa" actually calls a non-persisting validation endpoint OR the preview shows only the user-supplied values (ambiente + ptoVta) while CUIT/expiry are shown after server confirmation. Per CONTEXT.md: "preview modal appears showing extracted CUIT, cert expiry date, and selected ambiente — Admin confirms before anything is saved". This means the preview must be server-derived. Implementation: clicking "Vista previa" triggers `saveCert.mutateAsync()` — but NO: that would save. Re-read CONTEXT.md carefully: "After the Admin submits the form, a preview modal appears... Admin confirms before anything is saved." This implies the preview is a local preview of what WILL be submitted — since CUIT is extracted server-side, design this as: show the form → user clicks "Confirmar" (not "Vista previa") → this directly calls saveCert → on success the status view shows the confirmed data. The preview modal shows the user-entered values (ambiente prominent) + a warning that CUIT will be extracted from the cert. This is consistent with "confirm before saving" — the confirmation is the modal with the submitted values, and the actual API call happens on "Confirmar" in the modal.

So the flow is: fill form → click "Guardar" → modal opens with { certPem summary (first 50 chars), ptoVta, ambiente (PROMINENT — use large badge) } + "CUIT será extraído automáticamente del certificado" note → "Confirmar y guardar" button → `useSaveCert.mutate()` → on success: close modal, `isEditingCert = false`, toast "Certificado guardado correctamente". On error: close modal, toast with error message from API.

Loading state: while `saveCert.isPending`, show spinner + text "Validando con AFIP..." in the modal footer instead of the button.

**Section 2 — Configuración de facturación** (always visible when configured OR when in edit mode):
- ptoVta number input + ambiente select
- Pre-filled with current status values when configured
- "Guardar configuración" button → `useSaveBillingConfig.mutate({ ptoVta, ambiente })` → toast on success/error
- Loading state: "Validando con AFIP..." while pending
