<objective>
Cierra Gap B de 52-UAT (Test 13): un paciente que ya generó su portal link no puede re-mostrarlo,
porque por D-12 sólo se persiste el hash SHA-256 del token y el raw UUID se descarta tras la primera
generación (generarPortalLink devuelve url:null). El share panel queda atascado en la pantalla de
"Generar link" y el link se pierde.

Resolución decidida por el usuario (amplía D-12, no la re-litiga): almacenar el token crudo cifrado
con AES-256-GCM reutilizando `EncryptionService` (mismo helper de los WABA access tokens, usa
ENCRYPTION_KEY). El link se vuelve estable y recuperable; los lookups en BD siguen usando el hash
SHA-256 (portalToken). El raw token nunca se guarda en claro ni se loguea.

Este plan cubre SÓLO el backend (prisma → service → controller). El frontend se cierra en 52-10.

Purpose: Hacer el portal link recuperable entre sesiones sin romper D-12 (lookups por hash) ni
las defensas CR-01/T-52-01 del path de email.
Output: Columna cifrada + migración, generarPortalLink que recupera url, GET portal-link de sólo
lectura, EncryptionService inyectado en PacientesService, y unit tests del round-trip.
</objective>
