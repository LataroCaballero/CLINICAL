<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| paciente (browser, JWT portal) -> POST /paciente-portal/public/consulta | Input no confiable del paciente cruza al backend; el JWT porta el pacienteId |
| POST /consulta -> tabla MensajeInterno (chat interno del staff) | Texto del paciente se persiste y luego se renderiza en la UI del staff |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-01 | Elevation / Tampering | POST /consulta pacienteId | mitigate | pacienteId se toma de `req.user` (JWT portal-scoped), NUNCA del body; DTO narrow no declara pacienteId (D-03, pitfall 12) |
| T-55-02 | Tampering (mass-assignment) | CreateConsultaPortalDto | mitigate | `new ValidationPipe({ whitelist: true })` per-ruta descarta cualquier campo extra (autorId, prioridad, pacienteId); no hay pipe global, el pipe per-ruta es load-bearing |
| T-55-03 | Elevation (auth bypass) | @Post('consulta') | mitigate | `@UseGuards(PortalJwtGuard)` per-ruta obliga JWT scope=portal-paciente; guard nunca a nivel de clase para no romper rutas publicas |
| T-55-04 | Denial of Service (flood) | POST /consulta | accept | La clase ya tiene `@Throttle(20/min)` + ThrottlerGuard global (100/60s) heredado; suficiente para SC de esta fase |
| T-55-05 | Injection (stored-XSS) | mensaje persistido -> render en chat staff | mitigate | El texto se almacena como dato (Prisma parametrizado, sin SQL raw); el render seguro es responsabilidad del renderer del chat staff (React escapa por defecto; NO usar dangerouslySetInnerHTML). MaxLength(2000) acota el payload |
| T-55-SC | Tampering (supply chain) | npm installs | accept | Esta fase NO agrega dependencias npm nuevas (usa class-validator/prisma/nestjs ya presentes); sin superficie de supply-chain nueva |
</threat_model>
