<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| paciente (browser publico) -> portal/[token] page | Ruta publica sin login; el token crudo viaja en la URL |
| portal-api client -> backend paciente-portal | El JWT de portal se adjunta desde sessionStorage a cada request autenticado |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-06 | Information Disclosure | JWT de portal en el cliente | mitigate | El portalToken se guarda en `sessionStorage["portalToken"]`, SEPARADO del `localStorage.accessToken` del staff; el interceptor de portal-api solo adjunta el portalToken, nunca el accessToken staff; se limpia con clearPortalToken |
| T-55-07 | Spoofing / Elevation | portal-api en 401 | mitigate | El client NO reusa el interceptor de refresh/redirect a /login del staff; un 401 re-gatea por DNI, evitando fugas hacia el flujo de staff |
| T-55-08 | Denial of Service (brute force DNI) | DNI-gate | mitigate | El lock 3-intentos/15-min es server-side (F54); el cliente solo refleja el 429 con mensaje humano — no introduce superficie nueva de fuerza bruta client-side |
| T-55-09 | Injection (stored/reflected XSS) | render de datos del paciente / indicacionesUrl | mitigate | Render solo como texto (React escapa por defecto); prohibido dangerouslySetInnerHTML; NO renderizar indicacionesUrl como HTML hasta resolver CR-01 |
| T-55-SC | Tampering (supply chain) | npm installs | accept | Sin dependencias npm nuevas (axios, @tanstack/react-query, sonner, lucide-react ya presentes) |
</threat_model>
