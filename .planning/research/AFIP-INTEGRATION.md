## 1. Certificates (Per-Tenant / Per-CUIT)

### Certificate Types

AFIP requires an X.509 certificate linked to a CUIT (Argentine tax ID) for every web service call. Two certificate types:

- **Self-signed** — generated by the operator. Acceptable for homologacion and some production scenarios.
- **AFIP CA-issued** — issued via AFIP's "Administrador de Certificados Digitales" portal with fiscal key. Required for full production authorization in most cases.

### Homologacion vs Production

| Environment | How to get the cert | Portal |
|---|---|---|
| Homologacion (testing) | Generate a CSR, submit to **WSASS** (Web Service de Autenticación y Autorización Self-Signed) | `https://wsass-homo.afip.gov.ar/wsass/` |
| Produccion | Submit CSR to **Administrador de Certificados Digitales** with fiscal key level 3 | `https://auth.afip.gob.ar/contribuyente_/login.xhtml` |

### Storage Approach: AES-256-GCM (Model from EncryptionService)

> **v1.2 TODO:** Implement the encrypted columns below. The schema fields and the encryption approach are fully specified here — no re-research required.

Certificate and private key storage MUST follow the same AES-256-GCM pattern used for WABA access tokens. See `EncryptionService` at `backend/src/modules/whatsapp/crypto/encryption.service.ts`.

**Encryption format:** `iv:authTag:ciphertext` — all three segments Base64-encoded and joined with `:`. This is produced by `EncryptionService.encrypt()` and consumed by `EncryptionService.decrypt()`.

**Fields to add to `Profesional` or a dedicated `ConfiguracionAFIP` model in v1.2:**


**Security rule:** Never include `certPemEncrypted` or `keyPemEncrypted` in any DTO response. Decrypt only server-side at the time of the WSAA call.

### Provisioning Flow (Onboarding New Professional)

1. Professional (or their accountant) generates a CSR using their CUIT as CN.
2. Submits the CSR to AFIP's portal (homologacion or production, as appropriate).
3. Downloads the issued `.pem` certificate.
4. ADMIN uploads the `.pem` cert file and `.pem` private key file via a secure upload endpoint.
5. Backend receives them, calls `EncryptionService.encrypt(certPem)` and `EncryptionService.encrypt(keyPem)`.
6. Stores the two encrypted strings and `certExpiresAt` parsed from the cert's `notAfter` field.

### Key Rotation

Store `certExpiresAt` alongside the encrypted cert. Implement a scheduled job or check on WSAA call that warns when `certExpiresAt < now + 30 days`. Certificates typically expire after 2 years; failure to renew causes all AFIP calls to fail for that tenant.

## 2. WSAA (Authentication)

### What WSAA Does

Every AFIP web service call requires a valid Access Ticket (TA). The TA is bound to:
- A specific X.509 certificate (and therefore a specific CUIT)
- A specific target service (e.g., `wsfe`)
- A 12-hour validity window

### Endpoints

| Environment | Endpoint |
|---|---|
| Homologacion | `https://wsaahomo.afip.gov.ar/ws/services/LoginCms` |
| Produccion | `https://wsaa.afip.gov.ar/ws/services/LoginCms` |

Both are SOAP 1.1 endpoints accepting a `LoginCms(in0: string)` call where `in0` is the Base64-encoded CMS/PKCS#7 signed TRA.

### TRA XML Structure

The Token Request Authorization (TRA) is an XML document that MUST be signed before sending to WSAA:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<loginTicketRequest version="1.0">
  <header>
    <uniqueId>{unix_timestamp_seconds}</uniqueId>
    <generationTime>{ISO8601_UTC_minus_3_offset}</generationTime>
    <expirationTime>{generationTime_plus_12h}</expirationTime>
  </header>
  <service>wsfe</service>
</loginTicketRequest>
```

**Field rules:**
- `uniqueId`: Unix timestamp in seconds (integer). AFIP uses this to detect duplicate requests — use `Math.floor(Date.now() / 1000)`.
- `generationTime`: ISO 8601 with Argentina UTC-3 offset, e.g. `2026-03-13T09:00:00-03:00`. AFIP validates this against its own clock — see Clock Skew warning below.
- `expirationTime`: `generationTime + 12 hours`. AFIP will reject TAs after this time.
- `service`: The exact AFIP service name. For electronic invoicing: `wsfe`.

### TypeScript: `buildTRA()`

```typescript
// Source: pattern derived from afipjs documentation + WSAA technical spec
// Location (Phase 9): backend/src/modules/finanzas/afip/wsaa.service.ts

function buildTRA(service: string): string {
  const now = new Date();
  const exp = new Date(now.getTime() + 12 * 60 * 60 * 1000);

  // Argentina is UTC-3, no DST
  const toArgentineISO = (d: Date): string => {
    const offset = -3 * 60; // -180 minutes
    const localDate = new Date(d.getTime() + offset * 60 * 1000);
    return localDate.toISOString().replace('Z', '-03:00');
  };

  return `<?xml version="1.0" encoding="UTF-8"?>
<loginTicketRequest version="1.0">
  <header>
    <uniqueId>${Math.floor(now.getTime() / 1000)}</uniqueId>
    <generationTime>${toArgentineISO(now)}</generationTime>
    <expirationTime>${toArgentineISO(exp)}</expirationTime>
  </header>
  <service>${service}</service>
</loginTicketRequest>`;
}
```

### CMS/PKCS#7 Signing

The TRA must be signed as a CMS SignedData structure (PKCS#7), DER-encoded, then Base64-encoded. Node.js `crypto` does NOT natively produce PKCS#7 SignedData — this is not a simple `createSign()` call.

> **v1.2 TODO:** Choose and implement one of the two options below. Flag this as a team decision before Phase 9 implementation.

**Option A — Primary (recommended): `openssl smime -sign` subprocess**

```typescript
import { execSync } from 'child_process';
import * as fs from 'fs';
import * as os from 'os';
import * as path from 'path';

function signTRA(traXml: string, certPem: string, keyPem: string): string {
  // Write cert and key to temp files (they are in-memory after decrypt)
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'afip-'));
  const certPath = path.join(tmpDir, 'cert.pem');
  const keyPath = path.join(tmpDir, 'key.pem');
  const traPath = path.join(tmpDir, 'tra.xml');
  const outPath = path.join(tmpDir, 'tra.cms');

  try {
    fs.writeFileSync(certPath, certPem, { mode: 0o600 });
    fs.writeFileSync(keyPath, keyPem, { mode: 0o600 });
    fs.writeFileSync(traPath, traXml);

    execSync(
      `openssl smime -sign -signer ${certPath} -inkey ${keyPath} ` +
      `-in ${traPath} -out ${outPath} -outform DER -nodetach`,
      { stdio: 'pipe' },
    );

    const der = fs.readFileSync(outPath);
    return der.toString('base64');
  } finally {
    // Always clean up temp files
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
}
```

**Pros:** No new npm dependency. Works with any cert/key pair. `openssl` binary is standard on Linux servers.
**Cons:** Requires `openssl` binary on the host. Spawning a subprocess has a small latency cost (~20ms). Must clean up temp files.

**Option B — Alternative: `node-forge`**

```typescript
import * as forge from 'node-forge'; // npm install node-forge @types/node-forge

function signTRA(traXml: string, certPem: string, keyPem: string): string {
  const cert = forge.pki.certificateFromPem(certPem);
  const key = forge.pki.privateKeyFromPem(keyPem);

  const p7 = forge.pkcs7.createSignedData();
  p7.content = forge.util.createBuffer(traXml, 'utf8');
  p7.addCertificate(cert);
  p7.addSigner({
    key,
    certificate: cert,
    digestAlgorithm: forge.pki.oids.sha256,
    authenticatedAttributes: [
      { type: forge.pki.oids.contentType, value: forge.pki.oids.data },
      { type: forge.pki.oids.signingTime, value: new Date() },
      { type: forge.pki.oids.messageDigest },
    ],
  });
  p7.sign();

  const der = forge.asn1.toDer(p7.toAsn1()).getBytes();
  return Buffer.from(der, 'binary').toString('base64');
}
```

**Pros:** Stays in-process. No subprocess or temp files. Works on any OS.
**Cons:** Adds `node-forge` dependency (~800KB). Less battle-tested against AFIP's specific CMS format requirements.

### Token Extraction from LoginTicketResponse

The WSAA response XML contains a `loginCmsReturn` element with an embedded `LoginTicketResponse`:

```xml
<loginTicketResponse version="1.0">
  <header>
    <source>CN=wsaahomo, O=AFIP, C=AR, SERIALNUMBER=CUIT 33693450239</source>
    <destination>CN=your-cn, SERIALNUMBER=CUIT {your-cuit}</destination>
    <uniqueId>1773406674</uniqueId>
    <generationTime>2026-03-13T09:00:00-03:00</generationTime>
    <expirationTime>2026-03-13T21:00:00-03:00</expirationTime>
  </header>
  <credentials>
    <token>TOKEN_STRING_HERE</token>
    <sign>SIGN_STRING_HERE</sign>
  </credentials>
</loginTicketResponse>
```

Extract `<token>` and `<sign>` from `<credentials>`, and `<expirationTime>` for cache TTL. Use `xml2js` or a simple regex for these two values.

```typescript
// Simple extraction without full xml2js dependency (for Phase 9 stub):
function extractCredentials(loginTicketResponseXml: string): { token: string; sign: string; expiration: Date } {
  const tokenMatch = loginTicketResponseXml.match(/<token>([\s\S]*?)<\/token>/);
  const signMatch = loginTicketResponseXml.match(/<sign>([\s\S]*?)<\/sign>/);
  const expirationMatch = loginTicketResponseXml.match(/<expirationTime>([\s\S]*?)<\/expirationTime>/);

  if (!tokenMatch || !signMatch || !expirationMatch) {
    throw new Error('WSAA response missing required fields');
  }

  return {
    token: tokenMatch[1].trim(),
    sign: signMatch[1].trim(),
    expiration: new Date(expirationMatch[1].trim()),
  };
}
```

### Token Caching Strategy

```typescript
interface AccessTicket {
  token: string;
  sign: string;
  expiration: Date;
}

// Cache keyed by `${cuit}:${service}`
const ticketCache = new Map<string, AccessTicket>();

async function getAccessTicket(
  cuit: string,
  service: string,
  certPem: string,
  keyPem: string,
): Promise<AccessTicket> {
  const cacheKey = `${cuit}:${service}`;
  const cached = ticketCache.get(cacheKey);
  const fiveMinBuffer = 5 * 60 * 1000;

  if (cached && cached.expiration.getTime() - Date.now() > fiveMinBuffer) {
    return cached;
  }

  // Refresh: build TRA → sign → call WSAA → parse → cache
  const tra = buildTRA(service);
  const cms = await signTRA(tra, certPem, keyPem);
  const responseXml = await callWsaa(cms);
  const ticket = extractCredentials(responseXml);

  ticketCache.set(cacheKey, ticket);
  return ticket;
}
```

**Concurrency:** Use a per-CUIT mutex or lock to prevent multiple concurrent token renewals. Without a lock, two simultaneous requests for the same CUIT will each call WSAA, which is wasteful but not fatal. In v1.2, implement a proper mutex (e.g., `async-mutex` package or Redis-based lock if multi-instance).

### Clock Skew Warning

WSAA validates `generationTime` against its own clock. **The server must use NTP synchronization.** Clock drift of more than a few seconds causes WSAA to reject the TRA with:
- `"Fecha de generación inválida"` (generation time too far in the past or future)
- `"CMS: Error verificando firma"` (less specific, but often clock-related)

The 12-hour token window provides buffer for minor drift, but server clock must stay within a few seconds of true UTC.

### CRITICAL: Advisory Lock for CAE Sequencing

> **This is a correctness requirement, not an optimization. Missing it causes duplicate sequence errors in production.**

AFIP validates that `CbteDesde = FECompUltimoAutorizado + 1`. If two concurrent requests for the same professional + point-of-sale + invoice type submit simultaneously, both may query `FECompUltimoAutorizado` and get the same last number, then both attempt to submit with the same sequence number. AFIP will reject both (or accept one and reject the other with a duplicate error).

**Required pattern for v1.2:**

```typescript
// In the AfipService (Phase v1.2 real implementation):
async function emitirComprobante(params: EmitirComprobanteParams): Promise<EmitirComprobanteResult> {
  // Advisory lock key: hash of profesionalId + ptoVta + cbteTipo
  // This ensures only one CAE request per (profesional, ptoVta, cbteTipo) combination runs at a time
  await prisma.$executeRaw`
    SELECT pg_advisory_xact_lock(hashtext(${params.profesionalId} || ${params.ptoVta}::text || ${params.cbteTipo}::text))
  `;

  // Now safely: get last → build request → submit
  const lastAuthorized = await callFECompUltimoAutorizado(params.ptoVta, params.cbteTipo, auth);
  const cbteDesde = lastAuthorized + 1;
  const cbteHasta = cbteDesde; // single invoice

  return callFECAESolicitar({ ...params, cbteDesde, cbteHasta }, auth);
  // pg_advisory_xact_lock releases automatically when transaction commits or rolls back
}
```

The `pg_advisory_xact_lock` releases automatically at transaction end. This requires the entire `FECompUltimoAutorizado → FECAESolicitar` sequence to run within a single Prisma transaction (`prisma.$transaction`).

## 4. CAEA (Contingency Mode)

### What CAEA Is

CAEA (Código de Autorización Electrónico Anticipado) is an authorization code obtained in advance, valid for a 15-day billing period. AFIP issues one CAEA per period per taxpayer.

Periods:
- **First half:** Days 1–15 of the month
- **Second half:** Days 16–end of month

### RESTRICTION FROM JUNE 2026 (RG 5782/2025)

> **CRITICAL: CAEA is contingency-only from June 2026.**

RG 5782/2025 restricts CAEA use to contingency scenarios only, effective June 2026. The primary invoicing flow MUST be CAE (online, real-time). Using CAEA as a primary path is not permitted after this date.

**Design requirement for v1.2:** Always attempt CAE first. Only use CAEA when the AFIP `FECAESolicitar` endpoint is unavailable (network timeout, AFIP maintenance, HTTP 5xx).

> **Note:** RG 5782/2025 was found via community sources as of March 2026. **Verify against official BOLETIN OFICIAL before implementing CAEA fallback in v1.2.**

### Pre-Request Timing

CAEA must be requested within the **5 calendar days prior to the period start**:
- For the first-half period (days 1–15): request between the 27th–31st of the previous month.
- For the second-half period (days 16–end): request between the 11th–15th of the current month.

> **v1.2 TODO:** Implement a cron job that runs on days 27, 11, and 12 of each month to pre-request the CAEA for the next period. Use `@nestjs/schedule` (already a common NestJS pattern). Example cron: `0 6 27,11,12 * *` (6 AM UTC-3 on those dates).

### Contingency Flow

When CAE endpoint fails (timeout or error), the system uses a pre-fetched CAEA:

1. **Pre-request phase (cron):** `FECAEASolicitar(Periodo: 'YYYYMM', Orden: 1 or 2)` → store `{caea, fchVigDesde, fchVigHasta}` in the database against the professional's CUIT.
2. **Contingency mode:** When `FECAESolicitar` fails, assign the stored `caea` code to the invoice. Set `CbteAsoc` appropriately.
3. **Inform phase (within 30 days after period end):** Call `FECAEAInformar` with all invoices issued under CAEA during the period.

### `FECAEASolicitar` Key Fields

```xml
<FECAEASolicitar>
  <Auth>...</Auth>
  <Periodo>{YYYYMM}</Periodo>
  <Orden>{1 or 2}</Orden>
</FECAEASolicitar>
```

**Response fields:**
- `CAEA`: The authorization code string (store encrypted or at minimum restricted-access)
- `FchVigDesde`: Period start date YYYYMMDD
- `FchVigHasta`: Period end date YYYYMMDD
- `FchTopeInf`: Last date to inform invoices issued under this CAEA

### Production Reliability

AFIP production downtime is real. Known patterns:
- Scheduled maintenance windows (often announced on AFIP website)
- Fiscal year-end congestion (late December / early January)
- Month-end congestion when many businesses submit simultaneously

**Recommendation:** Always maintain a pre-requested CAEA for the current period as insurance, even in the CAE-primary flow. The cost is one extra AFIP call per period; the benefit is zero downtime for billing.

## 5. RG 5616/2024 Mandatory Fields

### Publication and Effective Dates

| Event | Date |
|---|---|
| Published in Boletín Oficial | December 18, 2024 |
| In force (non-exclusive) | April 15, 2025 |
| Non-exclusivity ends | March 31, 2026 |
| Full enforcement (rejections start) | April 1, 2026 |

From April 15, 2025: WSFEv1 accepts invoices without the new fields (non-exclusive period).
From April 1, 2026: WSFEv1 rejects invoices that omit the mandatory new fields.

### Mandatory New Fields in WSFEv1

**1. `CondicionIVAReceptorId`**
- The recipient's VAT status as an AFIP integer ID (see Section 3 mapping table).
- Mandatory for all electronic invoices from April 15, 2025.
- For B2C retail transactions (no CUIT): use `5` (Consumidor Final).
- The `Factura.condicionIVAReceptor` enum field added in Phase 8 (`facturador_v1` migration) stores this value.

**2. `MonCotiz` (for foreign currency invoices)**
- The BNA (Banco Nación Argentina) selling exchange rate (`tipo vendedor divisa`) for the business day prior to invoice issuance.
- Required when `MonId != "PES"` (i.e., for USD or other foreign currency invoices).
- For ARS invoices (`MonId = "PES"`): `MonCotiz = 1.00`.
- The `Factura.tipoCambio` field (Decimal 10,4) added in Phase 8 stores this rate.

### BNA Rate Source

The BNA does not have an official public API. Options for v1.2:
1. **Manual entry:** FACTURADOR user enters the BNA rate when creating a USD invoice. Display a link to `https://www.bna.com.ar/Personas` for reference.
2. **Automated scraping:** Scrape the BNA website (fragile, may break). Consider using a third-party currency rate service that sources from BNA.
3. **Hybrid:** Default to manual entry; optionally add an automated fetch as enhancement.

**Recommendation for v1.2:** Start with manual entry. It's robust, audit-friendly (user explicitly confirmed the rate), and doesn't add a scraping dependency.

## 6. Recommended Library Approach

### Decision: Raw SOAP/XML

**Use raw SOAP/XML — no third-party AFIP library.** This is a locked decision.

**Rationale:**
- `afipjs`, `afip-apis`, `afip.js` forks are unmaintained (last commits 3+ years ago, no TypeScript support, low star count, no active maintainer).
- Third-party libraries hide the WSAA token flow, making debugging harder when AFIP rejects a request.
- AFIP uses non-standard WS-Security variations that cause `soap` npm package to fail on specific endpoints.
- The WSAA and WSFEv1 APIs are stable and simple — raw SOAP is the right level of abstraction.

### What to Use

| Need | Tool | Status in Project |
|---|---|---|
| HTTP requests (SOAP calls) | `axios` | Already installed |
| XML parsing (WSAA response, WSFEv1 response) | `xml2js` or manual regex | `xml2js` not yet installed; manual regex sufficient for Phase 9 stub |
| CMS/PKCS#7 signing (TRA) | `openssl smime -sign` subprocess (primary) or `node-forge` (alternative) | Neither installed — v1.2 decision |
| Certificate encryption/decryption | Existing `EncryptionService` (AES-256-GCM) | Already available at `backend/src/modules/whatsapp/crypto/encryption.service.ts` |

### SOAP Envelope Pattern

All WSFEv1 calls follow this SOAP 1.1 envelope structure:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<soapenv:Envelope
  xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"
  xmlns:ar="http://ar.gov.afip.dif.FEV1/">
  <soapenv:Header/>
  <soapenv:Body>
    <ar:{MethodName}>
      <ar:Auth>
        <ar:Token>{token}</ar:Token>
        <ar:Sign>{sign}</ar:Sign>
        <ar:Cuit>{cuit}</ar:Cuit>
      </ar:Auth>
      <!-- method-specific parameters -->
    </ar:{MethodName}>
  </soapenv:Body>
</soapenv:Envelope>
```

The WSAA `LoginCms` envelope uses namespace `http://wsaa.view.sua.dvadac.desein.afip.gov` instead.

## Appendix: Open Questions for v1.2

1. **CMS signing approach** — `openssl smime` subprocess vs `node-forge`. Recommend team decision before Phase v1.2 starts. Both approaches are fully documented in Section 2.

2. **BNA rate source** — Manual entry (robust, recommended) vs automated scraping (fragile). See Section 5.

3. **`CondicionIVA` enum completeness** — 11 values confirmed from official AFIP documentation as of March 2026. AFIP may add values in future regulations. Adding Prisma enum values requires a migration but is non-breaking.

4. **CAEA pre-request cron holiday handling** — AFIP availability during the 5-day pre-request window on holidays is not documented. Recommend retry logic with 24-hour window.

5. **RG 5782/2025 verification** — The CAEA contingency-only restriction from June 2026 was found via community sources only. **Verify against official BOLETIN OFICIAL before implementing CAEA fallback in v1.2.**

6. **Multi-instance token cache** — The in-memory `Map<string, AccessTicket>` cache (Section 2) works for single-instance deployments. For multi-instance (horizontal scaling), use Redis for token cache to avoid redundant WSAA calls.
