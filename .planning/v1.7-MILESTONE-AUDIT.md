### Integration Issues Found

#### Issue 1 — rechazar() missing forward-only guard (Priority: High)

**Location:** `backend/src/modules/presupuestos/presupuestos.service.ts` line ~314

**Description:** The staff-triggered `rechazar()` method unconditionally writes `etapaCRM: EtapaCRM.PERDIDO` with no guard. The patient-triggered `rechazarByToken()` at line 648 correctly uses `etapasProtegidas = [CONFIRMADO, PROCEDIMIENTO_REALIZADO]` guard. A patient already confirmed for surgery can have their CRM stage forcibly reset to PERDIDO if staff clicks "rechazar" on an old budget.

**CRM-04 note:** CRM-04's explicit scope is "transiciones automáticas" (enviar/aceptar presupuesto). Staff-triggered rechazar may be outside that scope, but the behavioral asymmetry between rechazar/rechazarByToken appears unintentional.

**Fix:** Apply the `etapasProtegidas` pattern from `rechazarByToken()` to `rechazar()`.

#### Issue 2 — STEPPER_CHAIN vs ETAPA_ORDEN ordering conflict (Priority: Medium)

**Location:** `frontend/src/components/crm/EtapaStepper.tsx` vs `backend/src/modules/presupuestos/presupuestos.service.ts` / `turnos.service.ts`

**Frontend STEPPER_CHAIN order:**
```
SIN_CLASIFICAR(0) → NUEVO_LEAD(1) → TURNO_AGENDADO(2) → CONSULTADO(3)
→ PRESUPUESTO_ENVIADO(4) → PROCEDIMIENTO_REALIZADO(5) → CONFIRMADO(6)
```

**Backend ETAPA_ORDEN canonical order:**
```
SIN_CLASIFICAR(0) → NUEVO_LEAD(1) → TURNO_AGENDADO(2) → CONSULTADO(3)
→ PRESUPUESTO_ENVIADO(4) → CONFIRMADO(5) → PROCEDIMIENTO_REALIZADO(6)
```

**Impact:** The "Marcar como realizado" button at the PROCEDIMIENTO_REALIZADO step (index 5) appears visually *before* CONFIRMADO (index 6), contradicting the backend's canonical ordering where PROCEDIMIENTO_REALIZADO is *more advanced* than CONFIRMADO. A patient in CONFIRMADO seeing this button appears to be clicking "backward" in the stepper, even though the action is forward-only in the backend.

**Fix:** Align `STEPPER_CHAIN` in EtapaStepper.tsx to match backend `ETAPA_ORDEN` (swap CONFIRMADO and PROCEDIMIENTO_REALIZADO positions).

#### Issue 3 — Warning false positive edge case (Priority: Low)

**Location:** `backend/src/modules/pacientes/pacientes.service.ts` (getKanban) + `frontend/src/lib/crm-warnings.ts`

**Description:** `getKanban` returns the most recently *created* budget (`orderBy: { createdAt: 'desc' }, take: 1`). `getEtapaWarning` checks `patient.presupuesto?.estado !== 'ACEPTADO'` for the CONFIRMADO warning. Edge case: if a patient has budget_A (ACEPTADO, older) and budget_B (ENVIADO, newer), the returned budget is budget_B, triggering a false CRM-03 warning even though budget_A was accepted.

**Impact:** UX only — the move still proceeds. No data corruption.

**Fix:** Either prioritize ACEPTADO budgets in getKanban query, or filter by status before `take: 1`.
